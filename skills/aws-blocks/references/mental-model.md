# Mental model: how AWS Blocks resolves "the same code" onto three faces

This page is the foundation for understanding *why it works* and *why it breaks*. For the fine-grained
API you use while implementing, see the bundled docs (`node_modules/@aws-blocks/blocks/docs/`).

## The core: Node.js Conditional Exports
The mechanism is not magic and not codegen — it is the **condition** in the `exports` field of
`package.json`. The same `import` statement points at a **physically different file** depending on the
condition active at resolution time. This happens at two layers.

### Layer 1: frontend ↔ backend separation (the project's aws-blocks workspace)
The `exports` of `aws-blocks/package.json`:
- `browser` / `react-server` / `import` → `./client.js` (the auto-generated typed RPC client, used by `src/`)
- `types` / `default` → `./index.ts` (the backend implementation itself, used by the dev server / Lambda)

→ So when `src/` does `import { api } from 'aws-blocks'` it gets the client, and when the server imports
the same name it gets the implementation. This is the foundation of **types matching without codegen**.
`client.js` is auto-regenerated by the dev server / build from the exports of `index.ts` (don't edit it by
hand).

### Layer 2: local / cdk / aws-runtime separation (each Block package)
Example: the `exports` of `@aws-blocks/bb-distributed-table/package.json`:
```json
"exports": { ".": {
  "browser":     "./dist/index.browser.js",
  "cdk":        { "types": "./dist/index.cdk.d.ts", "default": "./dist/index.cdk.js" },
  "aws-runtime": "./dist/index.aws.js",
  "types":       "./dist/index.mock.d.ts",
  "default":     "./dist/index.mock.js"
}}
```
| condition | Resolves to | What's inside (same public API, different implementation) |
|---|---|---|
| `default` (no condition specified) | `index.mock.js` | In-memory + persistence to `.bb-data/*`. For local development |
| `cdk` | `index.cdk.js` | Creates an `aws-cdk-lib` `Table` and `grantReadWriteData()`. get/put etc. are synthGuard stubs |
| `aws-runtime` | `index.aws.js` | Real DynamoDB CRUD via `DynamoDBDocumentClient`. For production Lambda |
| `browser` | `index.browser.js` | Client-side stub |

All three files share the **same public API**, so the code in `index.ts` only has to exist once.

## The "switches" that toggle the condition (where the implementation lives)
| Context | Switch | Location |
|---|---|---|
| Local dev `npm run dev` | `tsx watch` (**no condition**) → `default` = mock | the `dev` script in the root `package.json` |
| CDK synth `npm run sandbox`/`deploy` | `NODE_OPTIONS="--conditions=cdk"` + `cdk.json`'s `npx tsx -C cdk ...` → `cdk` | `@aws-blocks/core/dist/scripts/sandbox.js` / `deploy.js`, `cdk.json` |
| Lambda bundle | esbuild `--conditions aws-runtime` → `aws-runtime` | `@aws-blocks/core/dist/cdk/blocks-backend.js` |
| Leak prevention | `assertCdkConditionActive()` throws when `--conditions=cdk` is missing (prevents accidentally synthing the mock) | same `blocks-backend.js` |

How the CDK entry is chained: `cdk.json` → `aws-blocks/index.cdk.ts`'s
`BlocksStack.create({ backendCDKPath: index.ts, backendHandlerPath: index.handler.ts })`.
In other words, **the CDK layer re-reads the same `index.ts` under the `cdk` condition** to derive the
infrastructure, and `index.handler.ts` (`createLambdaHandler(() => import('./index.js'))`) becomes the
production runtime entry point.

## The IFC layer, Scope, and fullId
- **IFC layer = `aws-blocks/index.ts`**: Block instantiation and API definition consolidated into one
  file. The infrastructure is derived from this code (you don't write a separate IaC file).
- **Scope**: a Block namespace. You place Blocks under `new Scope('my-app')`.
- **fullId**: `scope/id` (e.g. `my-app/todos`). This is the **basis for AWS resource names and the
  local `.bb-data/` directory name**. So changing the ID (the 2nd argument) changes the identity of the
  resource (→ see the data-loss section in rules-and-gotchas.md).

## How to verify it yourself (prove it read-only)
You can confirm on the spot that the resolution target changes with the condition:
```bash
# Yields the mock version (no condition specified)
node --input-type=module \
  -e "import('@aws-blocks/bb-distributed-table').then(m=>console.log(m.DistributedTable.toString().slice(0,200)))"

# Prefix it to switch to the cdk / aws version
NODE_OPTIONS=--conditions=cdk         node --input-type=module -e "..."
NODE_OPTIONS=--conditions=aws-runtime node --input-type=module -e "..."
```
Further, if you diff the output of `NODE_OPTIONS=--conditions=cdk npx cdk synth`
(`cdk.out/*.template.json`) against `index.ts`, you can see that a single line of
`new DistributedTable(...)` generates a DynamoDB::Table.

## Files worth comparing side by side to deepen understanding
1. `aws-blocks/index.ts` (the one line you write) ↔ `node_modules/@aws-blocks/bb-distributed-table/dist/index.{mock,cdk,aws}.js` (the three faces) ↔ the same package's `package.json` `exports` (the routing table)
2. `cdk.json` (`-C cdk`) ↔ `@aws-blocks/core/dist/scripts/sandbox.js` (`NODE_OPTIONS=--conditions=cdk`) ↔ `@aws-blocks/core/dist/cdk/blocks-backend.js` (esbuild `aws-runtime`)
